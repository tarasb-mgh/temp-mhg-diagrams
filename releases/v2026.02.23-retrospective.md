# Release Retrospective — v2026.02.23

**Date of release**: 2026-02-23
**Duration of release process**: ~3 hours (16:00–19:15 UTC)
**Previous release**: v2026.02.11 (12 days prior)
**Scope**: 6 repositories, 157 commits, 7 user-facing features

## Release Summary

This was the largest release to date for MentalHelpGlobal, delivering Google OAuth authentication, responsive PWA layouts, a dedicated workbench application split, supervisor roles, and Redis-backed token persistence across 6 repositories. The release required 3 hotfix iterations before all production services reached a healthy state.

## What Went Well

1. **Structured release process**. Following the release branch → PR → tag → deploy pipeline caught issues before they reached users and provided traceability for every change.

2. **Automated deployments worked reliably**. Once code reached main, chat-frontend and chat-backend deployed automatically without manual intervention.

3. **Health endpoint as an early warning system**. The `/api/health` endpoint surfaced the Redis issue immediately during sanity checks, before any user reports.

4. **Incremental hotfixing**. The PR cycle was followed even under time pressure — every hotfix went through a branch, PR, review, and merge. No direct pushes to main.

5. **No user-facing downtime**. Redis fallback to PostgreSQL meant authentication continued working throughout the Redis debugging. Users were never impacted.

6. **Divergence reconciliation**. Proactively merging main → develop before cutting release branches prevented a class of integration errors.

## What Went Wrong

### Incident 1: Workbench frontend had no production deploy workflow

**Severity**: Medium
**Impact**: Workbench was not deployed to production on the initial release. Required a hotfix cycle.
**Detection**: Post-deploy sanity check noticed workbench-frontend only had CI, no deploy.

**Root cause**: When the monorepo was split, `chat-frontend` received a `deploy.yml` but `workbench-frontend` only had `ci.yml` with a dev deploy step. The chat-ci repo had a reusable `deploy-workbench-frontend.yml`, but nothing triggered it from the workbench-frontend repo.

**What should have happened**: The `deploy.yml` should have been part of the workbench-frontend repo setup during the monorepo split (spec 001). The tasks and checklist for that spec should have included "verify production deploy workflow exists for each split repo."

### Incident 2: Workbench prod environment missing secrets

**Severity**: Low
**Impact**: First workbench deploy attempt failed. Took ~10 minutes to identify and fix.

**Root cause**: The `github-repos.json` in chat-infra didn't include `workbench-frontend` as a deployable repo. The `setup-github.sh` provisioning script therefore never configured its prod environment. The dev environment was set up manually but prod was missed.

**What should have happened**: When `workbench-frontend` was created as a new repo, it should have been added to `github-repos.json` and the `setup-github.sh` script should have been re-run.

### Incident 3: Redis ETIMEDOUT — 45 minutes of debugging

**Severity**: High (time cost), Low (user impact)
**Impact**: Redis token persistence was non-functional for ~1 hour. No user-facing impact due to PostgreSQL fallback.
**Time to resolve**: ~45 minutes across 3 hotfix attempts.

**Root cause**: The deploy workflow used `--set-secrets="REDIS_HOST=redis-host:latest"` to inject the Redis IP from GCP Secret Manager. While Cloud Run accepted the secret reference and created the revision successfully, the runtime value caused `connect ETIMEDOUT` errors. The exact mechanism is unclear — the secret values in Secret Manager were correct (`10.224.39.11` / `6379`), and the same IP worked when injected as a plain env var via `--set-env-vars`.

**Debugging path** (for reference):
1. Checked Secret Manager values — correct
2. Added firewall rule for VPC connector CIDR → Redis port — no effect
3. Switched from VPC connector to Direct VPC Egress — no effect
4. Verified TCP connectivity from a VM on the same VPC — **succeeded** (proving the Redis instance was reachable)
5. Switched from `--set-secrets` to `--set-env-vars` — **immediately worked**

**Hypothesis**: Secret Manager refs may inject values with subtle encoding differences (trailing whitespace, null bytes, or encoding that confuses ioredis's hostname resolution), or there may be a timing issue where the secret is resolved after the Redis client initialises its TCP connection.

**Resolution**: Redis host/port are now stored as GitHub environment variables and injected via `--set-env-vars`. This is appropriate since they are not sensitive values.

### Incident 4: Branch protection workaround for chat-infra

**Severity**: Low
**Impact**: Required temporarily disabling `enforce_admins` to merge the release PR.

**Root cause**: chat-infra has `enforce_admins: true` in branch protection, requiring all status checks to pass even for admins. The release PR had no required status checks configured for the repo (it's an infra/scripts repo with no CI), but the protection setting still blocked the merge.

**What should have happened**: Either configure appropriate status checks for chat-infra, or adjust the branch protection to allow admin merges for repos without CI pipelines.

### Incident 5: Merge conflicts during release

**Severity**: Low
**Impact**: Added ~15 minutes to the release process.

**Details**: Two sets of conflicts were encountered:
- **client-spec PR #5**: `specs/001-split-workbench-app/` had add/add conflicts from concurrent spec restructuring. Resolved via rebase.
- **Divergence reconciliation**: chat-infra (1 file) and workbench-frontend (25 files) had conflicts when merging main → develop. Resolved by keeping develop's version.

**Root cause**: The previous release's hotfix commits on main were never back-merged to develop. Over 12 days, develop diverged significantly.

## Key Learnings

1. **Infrastructure changes need their own release cycle.** Bundling VPC/Redis/DNS provisioning with feature work (OAuth, PWA) created a blast radius that was too large. The Redis issue would have been caught in a standalone infra release.

2. **"It works in dev" is not sufficient for infrastructure.** The Redis VPC connector setup was validated in dev, but the production deploy workflow didn't preserve the configuration. Deploy workflow changes need the same integration testing as application code.

3. **Secret Manager is not always the right tool.** Using Secret Manager for non-sensitive infrastructure config (Redis IP addresses) introduced an unnecessary failure mode. Plain environment variables are simpler, debuggable, and sufficient for non-secret values.

4. **Health endpoints are invaluable.** The structured `/api/health` response with per-service status (database, redis, email, dialogflow) made the Redis issue immediately visible and verifiable after each fix attempt.

5. **Fallback mechanisms save users.** The PostgreSQL fallback for refresh tokens meant Redis could be completely broken for an hour with zero user impact. Defensive architecture pays off during incidents.
