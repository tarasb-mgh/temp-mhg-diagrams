# Implementation Plan: Automated Post-Release Backmerge

**Branch**: `016-auto-backmerge` | **Date**: 2026-02-23 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/016-auto-backmerge/spec.md`

## Summary

Implement a reusable GitHub Actions workflow in `chat-ci` that automatically
creates and merges a PR from `main` → `develop` after every release merge.
Clean backmerges are merged immediately; conflicted ones are labeled for
manual resolution. Deploy the caller workflow to all 6 repos with a
`main`/`develop` branch model.

## Design

### Reusable Workflow: `chat-ci/.github/workflows/backmerge.yml`

**Trigger**: `workflow_call` (called by consumer repos)

**Inputs**:
- `base_branch` (string, default: `develop`) — target branch for backmerge
- `head_branch` (string, default: `main`) — source branch for backmerge

**Steps**:

1. **Check divergence**: Compare `head_branch` with `base_branch`. If
   identical (ahead_by == 0), exit successfully with no action.

2. **Check for existing PR**: Search for open PRs with label `backmerge`
   from `head_branch` to `base_branch`. If found, update body and exit.

3. **Create PR**: Create PR from `head_branch` → `base_branch` with:
   - Title: `chore: backmerge main → develop (triggered by <sha>)`
   - Label: `backmerge`
   - Body: list of commits included in the backmerge

4. **Attempt merge**: Run `gh pr merge --merge --delete-branch=false`.
   - On success: PR is merged, workflow exits.
   - On failure (conflict): Add `conflicts` label, post comment with
     conflict details, leave PR open for manual resolution.

**Permissions**: `contents: write`, `pull-requests: write`

### Caller Workflow: per-repo `.github/workflows/backmerge.yml`

```yaml
name: Backmerge main → develop
on:
  push:
    branches: [main]

jobs:
  backmerge:
    uses: MentalHelpGlobal/chat-ci/.github/workflows/backmerge.yml@main
    permissions:
      contents: write
      pull-requests: write
```

### Edge Cases

| Scenario | Behavior |
|----------|----------|
| `main` == `develop` (no divergence) | No PR created, clean exit |
| Clean merge possible | PR created and merged immediately |
| Merge conflicts | PR created, labeled `conflicts`, comment posted |
| Existing backmerge PR open | Body updated, no duplicate created |
| Workflow triggered by backmerge merge | No-op (backmerge merges to develop, not main) |
| `develop` branch doesn't exist | Workflow fails gracefully with error message |

## Deployment Plan

1. Create `backmerge.yml` in `chat-ci` on a feature branch, merge to `main`
2. Create caller `backmerge.yml` in each of the 6 target repos
3. Verify by pushing a test commit to `main` in one repo and confirming
   the backmerge PR appears and merges

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Infinite loop (backmerge triggers itself) | Low | High | Backmerge merges to `develop`, not `main` — no loop possible |
| GITHUB_TOKEN insufficient permissions | Low | Medium | Verify token scopes in workflow permissions block |
| Branch protection added to `develop` later | Medium | Low | Workflow falls back gracefully; update to use `--auto` flag |
