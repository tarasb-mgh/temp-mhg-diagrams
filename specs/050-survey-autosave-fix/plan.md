# Implementation Plan: Survey Schema Autosave Fix + Regression Coverage

**Branch**: `050-survey-autosave-fix` | **Date**: 2026-04-14 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/050-survey-autosave-fix/spec.md`

## Summary

Fix three autosave bugs in the workbench survey schema editor (race condition, silent validation gate, ID sync) and add comprehensive regression test coverage. Root cause analysis identified that the `useDebouncedSave` hook treats saves as fire-and-forget without reconciling local state with server responses. Previous fixes (#107, #113, #119, #121, #124, #153) patched symptoms on the backend; this fix addresses the frontend root causes.

## Technical Context

**Language/Version**: TypeScript 5.x (React 18)

**Project Type**: Web application (monorepo split: workbench-frontend + chat-backend + client-spec)  
**Constraints**: 2-second debounce interval, <500ms save round-trip on dev, 30s conflict poll

## Implementation Phases

### Phase 1: Fix useDebouncedSave Hook (US1 + US2 + US3)

**Files to modify**: `workbench-frontend/src/features/workbench/surveys/hooks/useDebouncedSave.ts`

#### US1: Race Condition Fix

**Current code** (lines 53-87):
```typescript
const doSave = useCallback(async () => {
  if (!schemaId || isReadOnly || savingRef.current) return;  // ← blocks without queuing
  // ... save logic ...
  savingRef.current = false;
  if (result.success) {
    setHasUnsavedChanges(false);  // ← unconditionally clears, even if new edits occurred
  }
}, [...]);
```

**Fix approach**:
1. Add `pendingSaveRef = useRef(false)` to track whether a save was blocked
2. When `savingRef.current` is true, set `pendingSaveRef.current = true` instead of returning
3. After the in-flight save completes, check `pendingSaveRef` and schedule an immediate retry
4. Track a `saveSnapshotRef` (serialized state at save-start time); only clear `hasUnsavedChanges` if current state matches the snapshot

#### US2: Validation Gate Feedback

**Current code** (lines 56-66):
```typescript
if (!title || !title.trim()) return;
for (const q of questions) {
  if (!q.text || !q.text.trim()) return;  // ← silent return, no status change
}
```

**Fix approach**:
1. Add a new `SaveStatus` value: `'validation_blocked'`
2. When validation fails, set `setSaveStatus('validation_blocked')` instead of silently returning
3. Update `SaveStatusIndicator.tsx` to render a distinct visual for this state (e.g., amber dot + "Complete all questions to save")

#### US3: ID Sync After Save

**Current code** (lines 80-84):
```typescript
if (result.success && result.data) {
  lastSavedUpdatedAt.current = result.data.updatedAt;
  // ← Missing: sync question IDs back to local state
}
```

**Fix approach**:
1. Accept an `onSaveSuccess` callback in the hook options
2. In `SurveySchemaEditorView`, pass a callback that merges server-assigned question IDs into local state
3. The callback uses `setQuestions(prev => ...)` to update IDs without triggering a full re-render cycle
4. Guard against the merge itself triggering the debounce effect (use a ref flag to skip the next effect run)

## Risks

| Risk | Mitigation |
|------|-----------|
| ID sync callback triggers infinite re-render loop | Use `skipNextEffectRef` flag to break the cycle |
| Race condition fix introduces duplicate saves | Use `pendingSaveRef` boolean (not counter) — at most one queued retry |
| New SaveStatus value breaks existing consumers | Only `SaveStatusIndicator` reads `saveStatus` — contained blast radius |
| Regression tests flaky due to timing | Use explicit `browser_wait_for` with 3-4s delays (> 2s debounce) |
