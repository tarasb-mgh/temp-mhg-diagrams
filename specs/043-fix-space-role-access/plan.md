# Implementation Plan: Fix Workbench Space Selector Role-Based Access

**Branch**: `043-fix-space-role-access` | **Date**: 2026-04-02 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/043-fix-space-role-access/spec.md`

## Summary

The workbench Space selector incorrectly falls back to the first membership-based Space when a global role holder (Researcher, Supervisor, Moderator, Owner) selects a Space they are not a member of. This contradicts the two-tier access model from spec 034. The fix targets three layers: (1) backend `setActiveGroup()` must skip membership validation for global role holders, (2) backend `GET /api/admin/groups` must return all non-archived groups for global role holders, and (3) frontend `GroupScopeSelector.tsx` must not apply client-side fallback logic that overrides a successful backend response. No database schema changes required — all changes are application-level logic fixes in existing code.

## Implementation Order

### Phase 0: Root Cause Investigation

**Goal**: Confirm exact locations of the three suspected defects before writing any code.

1. Read `groupMembership.service.ts` → confirm whether `setActiveGroup()` calls membership validation for global role holders
2. Read `admin.groups.ts` → confirm whether `GET /api/admin/groups` filters groups by membership for global role holders
3. Read `GroupScopeSelector.tsx` → confirm whether client-side fallback logic resets dropdown selection

### Phase 1: Backend Fixes (chat-backend)

**Depends on**: Phase 0 (root cause confirmed)

4. Fix `groupMembership.service.ts` — in `setActiveGroup()`, add early return path that skips membership validation when `hasGlobalGroupAccess(userRole)` returns true; still validate group exists and `archived_at IS NULL`
5. Fix `admin.groups.ts` — ensure `GET /api/admin/groups` returns all non-archived groups when the requesting user has a global role >= Researcher; filter archived groups for all users
6. Add unit tests in `groupAccess.service.test.ts`:
   - Global role holder can set active group to a non-member group
   - QA Specialist cannot set active group to a non-member group
   - No user can set active group to an archived group
   - Role downgrade: user loses non-member group access on next request

### Phase 2: Frontend Fix (workbench-frontend)

**Depends on**: Phase 1 (backend returns correct responses)

7. Fix `GroupScopeSelector.tsx`:
   - For users with global role access: call group list endpoint that returns all groups (already available from Phase 1)
   - Remove or bypass any client-side fallback logic that resets dropdown to a membership-based group when backend returns 200
   - On server error during space switch: revert dropdown to previous active Space and show error notification (FR-012)
   - On role downgrade (backend returns 403 for active group): clear active group, show empty state (FR-013)

### Phase 3: Regression Verification

**Depends on**: Phase 1 + Phase 2

8. Manual verification on dev environment:
   - Log in as Researcher (zero memberships) → verify all groups in dropdown → select non-member group → verify content loads → refresh → verify persistence
   - Log in as QA Specialist → verify only membership groups shown
   - Log in as Owner → verify full access
9. Chat frontend regression: verify chat group selector unchanged for regular users
10. Verify no `canAccessGroup()` interface changes — inspect function signature matches spec 034 definition
