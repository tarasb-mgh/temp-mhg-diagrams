# Data Model: MCP Workbench Server

**Date**: 2026-04-15
**Feature**: 051-mcp-workbench-server

## Overview

The MCP server itself is stateless — it holds no persistent data. The entities below describe the transient runtime structures that exist in-memory (per MCP session) and in Redis (for device auth).

## Entities

### DeviceAuthorizationRequest

Represents a pending device authorization flow. Created when the MCP server initiates authentication and destroyed when the user completes login or the code expires.

| Field | Type | Description |
|-------|------|-------------|
| deviceCode | string | 256-bit hex token (server-side identifier for polling) |
| userCode | string | 8-char base-20 code (human-readable, displayed to user) |
| clientId | string | Always `"mcp-server"` for this integration |
| userId | string? | null until user authenticates; set to user UUID on auth completion |
| authorized | boolean | false → true when user completes browser login |
| scope | string | Space-separated permission scopes (e.g., `"workbench"`) |
| createdAt | ISO timestamp | When the request was initiated |
| expiresAt | ISO timestamp | createdAt + 15 minutes |
| pollInterval | number | Minimum seconds between token polls (default: 5) |
| failedAttempts | number | Count of invalid user_code entries (max: 5 before lockout) |

**Storage**: Redis with 900-second TTL (auto-expiry)
**Keys**:
- `device_code:{deviceCode}` → full object
- `user_code:{userCode}` → deviceCode (reverse lookup)

**State transitions**:
```
Created (authorized=false, userId=null)
  → User authenticates in browser → Authorized (authorized=true, userId=UUID)
  → MCP server polls and receives token → Consumed (deleted from Redis)
  → TTL expires → Expired (auto-deleted by Redis)
  → 5 failed attempts → Locked (failedAttempts=5, no further validation attempts accepted)
```

### McpSession

Represents a single SSE connection from a client. Exists in-memory on the Cloud Run instance handling the connection. Destroyed when the SSE connection closes.

| Field | Type | Description |
|-------|------|-------------|
| sessionId | string | Auto-generated by MCP SDK per SSE connection |
| userId | string | UUID of the authenticated user |
| email | string | User's email address |
| role | string | User's RBAC role (owner, moderator, researcher, etc.) |
| permissions | string[] | Effective permissions (e.g., `["REVIEW_ACCESS", "REVIEW_SUBMIT"]`) |
| accessToken | string | Current JWT access token |
| refreshToken | string | JWT refresh token for auto-renewal |
| tokenExpiresAt | ISO timestamp | When the access token expires |
| permissionsLoadedAt | ISO timestamp | When permissions were fetched (for staleness indicator) |
| connectedAt | ISO timestamp | When the SSE connection was established |

**Storage**: In-memory (per Cloud Run instance)
**Lifecycle**: Created on SSE connect + auth complete → Destroyed on SSE disconnect or instance shutdown

---

### ToolDefinition

Represents a registered MCP tool with its permission requirements. Defined at server startup (compile-time), not stored in a database.

| Field | Type | Description |
|-------|------|-------------|
| name | string | MCP tool name (e.g., `list_review_sessions`) |
| description | string | Human-readable tool description |
| inputSchema | ZodSchema | Zod schema defining accepted parameters |
| requiredPermissions | string[] | Permissions the user must hold to execute this tool |
| handler | function | Async function that calls the backend API and returns formatted results |
| tier | number | 1 or 2 — indicates feature tier for documentation purposes |
| domain | string | Tool domain (review, safety, users, surveys, dashboards, tags, groups, supervision) |

**Storage**: Compile-time constant (TypeScript source code)

---

### ToolInvocationContext

Passed to each tool handler at execution time. Provides the session context needed to make authenticated backend API calls.

| Field | Type | Description |
|-------|------|-------------|
| session | McpSession | The current user's session (role, permissions, tokens) |
| apiClient | ApiClient | Pre-configured HTTP client with the user's JWT |

---

### ApiClient

HTTP client used by tool handlers to call the chat-backend REST API.

| Field | Type | Description |
|-------|------|-------------|
| baseUrl | string | Backend API URL (from `BACKEND_API_URL` env var) |
| session | McpSession | Reference to the session for JWT injection and refresh |

**Behaviors**:
- Injects `Authorization: Bearer {accessToken}` on every request
- On 401 response: attempts token refresh via `POST /api/auth/refresh`, retries once
- On refresh failure: returns auth error prompting re-authentication
- Maps HTTP errors to MCP-friendly error objects

## Relationships

```
DeviceAuthorizationRequest  ──(1:1 on auth complete)──►  McpSession
McpSession  ──(1:1)──►  ApiClient
McpSession  ──(checked by)──►  ToolDefinition.requiredPermissions
ToolDefinition  ──(invoked with)──►  ToolInvocationContext { session, apiClient }
```

## Entities NOT Owned by MCP Server

The following entities are owned by the chat-backend. The MCP server reads/writes them via API calls but does not define or store them:

- **User** (email, role, status) — read via `GET /api/users`
- **ReviewSession** (messages, rating, tags) — read/write via `GET/POST /api/review`
- **SurveySchema** (title, questions, status) — read/write via `GET/POST /api/surveys/schemas`
- **SurveyInstance** (groups, dates, responses) — read/write via `GET/POST /api/surveys/instances`
- **RiskFlag** (severity, disposition, notes) — read/write via `GET/POST /api/risk/flags`
- **Group** (name, members) — read via `GET /api/groups`
- **ReviewTag** (name) — read/write via `GET/POST /api/admin/tags`

These are documented in their respective feature specs (002, 011, 018, 029, 032, etc.).
