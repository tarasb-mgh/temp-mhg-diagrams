# Implementation Plan: Global Role Group Access

**Branch**: `034-global-role-group-access` | **Date**: 2026-03-19 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/034-global-role-group-access/spec.md`

## Summary

Global role holders (Researcher, Moderator, Supervisor, Owner) need to access all group-scoped workbench functionality without explicit group membership. This feature introduces a two-tier access resolution model: a shared `canAccessGroup()` function that grants access if the user is a group member OR holds a global role >= Researcher. No database schema changes are required — all changes are application-level logic across `chat-types`, `chat-backend`, and `workbench-frontend`.

## Modified Endpoint Behaviors

No new endpoints are created. Existing endpoints are modified as follows:

### GET /api/admin/groups
**Current**: Requires `WORKBENCH_USER_MANAGEMENT` or `SURVEY_INSTANCE_MANAGE` or `SURVEY_INSTANCE_VIEW`
**Changed**: Also allows `WORKBENCH_GROUP_DASHBOARD` (which Researchers have)
**Response**: Unchanged — array of `GroupDto` objects. Archived groups already filtered by existing logic.

### GET /api/group/dashboard, GET /api/group/users
**Current**: Requires permission + resolves groupId from header; no explicit membership check beyond `resolveGroupId()`
**Changed**: `canAccessGroup()` enforced. Global role holders can access any group. Returns 403 `{ error: 'FORBIDDEN' }` if access denied.

### GET /api/admin/groups/:groupId/members
**Current**: Returns users with explicit group membership
**Changed**: No response change. Verified: global-role-only users never appear (they have no `group_memberships` record).

### POST /api/group/set-active
**Current**: Requires active membership in target group
**Changed**: Global role holders (Researcher+) can set any non-archived group as active without membership.

### Review/session/deanonymization endpoints
**Current**: OWNER or REVIEW_CROSS_GROUP bypasses membership check; others need membership
**Changed**: All global role holders (Researcher+) bypass membership check via `canAccessGroup()`. Returns 403 `{ error: 'FORBIDDEN' }` if denied.

### Audit log entries (all group-scoped endpoints)
**Current**: `details` JSONB has action-specific metadata
**Changed**: `details` gains `access_type: "membership" | "global_role"` for group-scoped actions by workbench users
