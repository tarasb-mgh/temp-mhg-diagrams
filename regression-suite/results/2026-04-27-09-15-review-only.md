**However**, three defects were observed in the queue / card data layer that warrant immediate triage:

1. **D-1: Dashboard ↔ Queue counter mismatch.** Dashboard reports `Pending Review: 91`. Review Queue → Pending tab shows `0` sessions. Cause confirmed via network: queue filters by selected Space (`groupId=ba1ced2d-...`, the "Dev team" space), Dashboard does not filter by space. The two counters answer different questions but appear identical to users — they should either both filter or both aggregate, with explicit labeling.

2. **D-2: "Broken" sessions surfaced in In Progress queue.** Both 2 sessions in In Progress are marked `Broken` with sub-status `UNFINISHED`, 47 days old, 0/3 reviews. Tooltip text says: *"This session has no assistant replies to rate — it was surfaced by mistake."* These should not be reachable from the reviewer queue without an explicit "show broken" filter.

3. **D-3: Session card tooltip contradicts session data.** The same "broken" tooltip claims "no assistant replies", yet opening session `be2dc550-...` shows 8 assistant messages (and 8 user messages) — a fully populated transcript. Either the session-broken detection logic is wrong, or the tooltip is shown for the wrong sessions.

The P0 path is clean enough to merit a full smoke run after these three defects are addressed.

## Defects (real bugs, not test issues)

### D-1 — Dashboard counter ≠ Queue count (Severity: Medium)
**Symptom:** Dashboard says `Pending Review: 91`. Review Queue → Pending tab shows `0`.
**Root cause (verified):** The `/api/review/queue` endpoint is called with `groupId=ba1ced2d-e827-4a2e-8d16-94fc3085962e` (the "Dev team" space, currently selected in the Space combobox). The Dashboard `Pending Review` counter does not filter by space.
**User impact:** Reviewer sees "91 pending" on the Dashboard, navigates to the queue, sees an empty list, and is confused. Either the click-through goes to a queue with the wrong scope, or the counter is misleading.
**Fix options:**
- Make Dashboard's "Pending Review" card respect the active Space selection.
- Or label the Dashboard counter as "Pending Review (All Spaces)" and add per-space breakdown.
- Or auto-switch Space to "All Spaces" when navigating from the Dashboard tile.

### D-2 — "Broken" sessions are reachable from Reviewer In Progress tab (Severity: High)
**Symptom:** Both sessions in In Progress (`CHAT-8611`, `CHAT-EC17`) display the `Broken` badge with hover tooltip *"This session has no assistant replies to rate — it was surfaced by mistake."*
**User impact:** A Reviewer opens a "broken" session and finds it un-rateable; productivity loss. The system already knows it was "surfaced by mistake" yet keeps showing it.
**Fix:** Filter `Broken` / `surfaced-by-mistake` sessions out of the queue, or add a banner on the queue explaining what they are.

### D-3 — Tooltip text contradicts session content (Severity: Medium)
**Symptom:** Tooltip says "no assistant replies"; the same session opens with 8 assistant messages.
**User impact:** Mistrust in the broken-session detection. Either it's catching the wrong sessions, or the tooltip text is wrong for the catch-criterion that fires.
**Fix:** Audit the criteria that mark a session "Broken" and reconcile with the tooltip wording. Likely candidates: the session has assistant messages but they're empty/null, or the message-count metric is computed from a different table than what the detail view reads.
